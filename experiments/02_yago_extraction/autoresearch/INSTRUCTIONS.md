# Autoresearch: mechanistic extraction of YAGO-UUID canaries

## Goal
Find a method that **extracts more memorized UUIDs from held-out canaries than plain greedy
decoding does**. The target model (`allegrolab/hubble-1b-100b_toks-perturbed-hf`) was trained on
synthetic biographies, each ending in `"<Name> has the unique identifier <32-hex-char UUID>"`. A
UUID is uniform-random, so reproducing it can only be memorization. Hubble inserted each biography
a known number of times (`duplicates`), giving us **ground-truth memorization strength** — that is
the signal you calibrate against.

The working hypothesis (why this is open): output-level attacks (greedy, prefix tuning, candidate
rerankers) read only the model's probabilities, which conflate "recalled a stored string" with
"made a confident guess". The memorization signal is **mechanistic** — in the activations/circuit,
not the logits. Promising directions, roughly in order:
1. **Read** — logit/tuned lens at the secret positions: is the correct token present in an
   intermediate layer even when the final output is wrong? (The greedy baseline derails after a few
   correct hex chars, which smells like a mid-stack recall signal that later layers overwrite.)
2. **Localize** — activation patching / causal tracing to find the layer/heads/neurons carrying the
   secret; look for a sparse component that spikes on memorized canaries vs. dup=0.
3. **Intervene** — amplify/steer/patch that component during decoding and measure held-out lift.

## Success criterion (read this before claiming anything)
For every method, the harness records, **per duplication level and per split**, two metrics:
- `exact` — verbatim extraction rate (the real goal).
- `token` — fraction of the UUID's tokens recovered (a sensitive early indicator; use it to detect
  progress while `exact` is still 0).

A method **wins** iff its **`held_exact` exceeds the greedy baseline's `held_exact`** at some
duplication level (use `held_token` as the leading indicator). Rules:
- **Train numbers never count as success.** High `train_*` with low `held_*` is overfitting (the
  prefix-tuning failure mode: it memorized the supervision instead of learning to extract). The
  `train − held` gap is the overfitting measure; report it honestly.
- **`dup=0` must stay at the floor (~0 exact, ~0.025 token) on both splits.** If it rises, the
  method is leaking the answer and the result is invalid.
- Compare against `results/greedy_baseline.json`, generated by running the baseline below.

## What is LOCKED vs. EDITABLE
**LOCKED — never edit:**
- `harness.py` — defines the split, the two metrics, and the JSON schema. This is the reward.
  Editing it (or moving scoring into your method) invalidates every result.

**EDITABLE — this is your workspace:**
- `methods/<name>.py` — add new methods here. Each exposes `build(model, tokenizer)` returning an
  object with `fit(train_records)` and `generate(records)` that attaches a candidate string at
  `record["generation"]`. You may use anything in `src/hubble/` and add helpers there too — but do
  **not** repurpose the harness's metrics.
- `run_method.py` — the driver (you can extend it, but it must route through `harness.evaluate`).
- `LOG.md` — record every attempt and its outcome (see below).

You innovate only on **how the candidate string is produced**. Whether it counts as an extraction
is decided in `harness.py` and nowhere else.

## How to run
A GPU is needed. If one is free (`nvidia-smi`), run directly; otherwise submit via slurm.
```
uv run python experiments/02_yago_extraction/autoresearch/run_method.py --method greedy_baseline
```
This writes `results/greedy_baseline.json` (the consistent record) and
`results/greedy_baseline_generations.jsonl` (raw generations for inspection). Run the baseline
first so you have the bar to beat.

## Workflow (do this every iteration)
1. Write `methods/<name>.py`.
2. Run it through the driver.
3. Read `results/<name>.json`; compare `held_exact` / `held_token` to the greedy baseline.
4. Check the overfitting gap and the `dup=0` floor.
5. Append a row to `LOG.md`: method, one-line idea, key held-out numbers, and verdict
   (win / no lift / overfit / invalid). Keep failed ideas — they're the map.
